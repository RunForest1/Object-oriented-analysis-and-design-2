# Система конвертации документов «DocBridge» с использованием паттерна Мост

---

### Назначение
В современных информационных системах часто возникает задача поддержки множества форматов данных (JSON, PDF, Markdown) для различных типов документов (текстовые отчеты, медиа-галереи, таблицы).

Традиционный подход с использованием наследования приводит к «взрыву количества классов»: для каждой комбинации типа документа и формата экспорта требуется свой класс (например, TextPdfDocument, TextJsonDocument, ImagePdfDocument, ImageJsonDocument). Это делает систему жесткой, сложной в поддержке и расширении. Добавление нового формата требует переписывания кода всех типов документов.

### Решение: Паттерн Мост (Bridge Pattern)
Идея заключается в разделении абстракции (типа документа) и реализации (формата экспорта), чтобы они могли изменяться независимо друг от друга.

- Абстракция (**Document**): Определяет высокоуровневую логику работы с документом (добавление элементов, хранение данных). Система имеет иерархию конкретных документов: **TextDocument** (работает только с текстом) и **ImageDocument** (работает с путями к изображениям).
- Реализация (**IExporter**): Определяет низкоуровневый интерфейс для преобразования данных в конкретный формат. Система имеет набор независимых реализаций: **JsonExporter**, **PdfExporter**, **MarkdownExporter**.
- Мост: Абстракция хранит ссылку на объект реализации и делегрует ему задачу экспорта.

# Выводы по лабораторной работе: Паттерн Мост (Bridge)

## Сравнительный анализ

| Критерий | С паттерном Мост (Bridge) | Без паттерна (Прямое наследование) |
| :--- | :--- | :--- |
| **Способ организации кода** | **Композиция:** Абстракция (`Document`) содержит ссылку на Реализацию (`IExporter`). Две независимые иерархии классов. | **Наследование:** Создание отдельного класса для каждой комбинации типа документа и формата (например, `TextJsonDoc`, `ImagePdfDoc`). |
| **Количество классов** | **Линейный рост ($N + M$):** 2 класса документов + 3 класса экспортеров = **5 классов**. | **Квадратичный рост ($N \times M$):** Для тех же условий требуется **6 классов**. При добавлении нового формата количество классов удваивается. |
| **Расширяемость** | **Высокая:** Добавление нового формата требует создания всего **одного** нового класса-экспортера. Он автоматически становится доступен всем типам документов. | **Низкая:** Добавление нового формата требует создания **новых классов для каждого типа документа** (например, добавить `TextXmlDoc` и `ImageXmlDoc`). |
| **Сохранение состояния** | **Автоматическое:** Данные хранятся в объекте Абстракции. Смена реализации (`setExporter`) не требует копирования данных, так как объект документа не пересоздается. | **Ручное:** Смена формата означает уничтожение старого объекта и создание нового. Требуется явное копирование данных из старого объекта в новый через временные переменные. |
| **Инкапсуляция** | **Высокая:** Логика форматирования (синтаксис JSON, PDF) полностью скрыта внутри классов-реализаций и не засоряет классы документов. | **Смешанная:** Логика хранения данных и логика их экспорта находятся в одном классе, что нарушает принцип единственной ответственности (SRP). |
| **Гибкость runtime** | **Динамическая:** Возможность менять реализацию (формат экспорта) прямо во время выполнения программы без потери данных. | **Статическая:** Тип объекта определяется в момент создания (`new`). Изменить поведение существующего объекта невозможно без его полной замены. |

## Итоговый вывод

1.  **Разделение ответственности:** Паттерн Мост позволил четко разделить «что делает система» (управление документами: текст/картинки) и «как она это делает» (форматирование: JSON/PDF/Markdown). Это сделало код чище и понятнее.
2.  **Защита от «взрыва классов»:** Мы избежали создания матрицы классов (`TextJson`, `TextPdf`, `ImageJson`...). Вместо этого мы создали две небольшие независимые иерархии, связанные через агрегацию.
3.  **Упрощение поддержки:** В реализации без паттерна нам пришлось писать сложный код с `GetData()` и циклами восстановления состояния при смене формата. В реализации с Мостом эта проблема решается одной строкой `doc->setExporter(...)`, так как данные живут отдельно от алгоритма экспорта.
4.  **Применимость:** Паттерн идеален для задач, где есть несколько независимых осей изменения (в нашем случае: *Тип контента* и *Формат вывода*), и когда необходимо менять поведение объекта динамически.
